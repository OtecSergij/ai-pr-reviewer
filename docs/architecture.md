# Архитектура и конвенции
## 1. Структура каталогов

- **`app/` — только роутинг и UI.** Страницы, layout-ы, route handlers, клиентские компоненты и хуки. Никакой доменной или инфраструктурной логики здесь не живёт.
- **`lib/` — вся доменная и инфраструктурная логика.** Фреймворк навязывает форму только `app/`; всё остальное организуем сами внутри `lib/`.
- **Граф зависимостей однонаправленный:** `app → components → lib`. `lib/` никогда не импортирует из `app/`.
- **Организация по доменам и границам, а не по техническим слоям.** Не заводим папки вида `services/`, `controllers/`, `models/` на верхнем уровне. Вместо этого: доминирующая фича получает единый дом, внешние сервисы прячутся за адаптерами, инфраструктура выделена отдельно.
- **У доменной сущности — один дом.** Все состояния и логика одной сущности лежат рядом, а не размазаны по нескольким техническим папкам.

Целевой каркас (ориентир на уровне папок, не жёсткий список):

```
app/            роутинг, UI, route handlers (тонкие)
lib/
  review/       единый дом фичи: доменная модель, схемы границ,
                контракт стрима, оркестрация, инструменты, промпт
  ai/           инфраструктура LLM (конфигурация провайдера)
  github/       адаптер внешнего сервиса (порт + реализация)
  env           валидация окружения
```

## 2. Границы и перевод данных

- **Внешние границы говорят на своём языке, домен — на своём.** Источники внешнего формата: GitHub API и схемы, обращённые к модели (LLM). Между ними и доменом всегда есть точка перевода.
- **Перевод происходит в адаптере, на входе в домен.** Внешний модуль отдаёт наружу уже доменный формат; домен не знает о форме внешнего ответа.
- **`snake_case` — язык границ, `camelCase` — язык домена и UI.**
- **Критерий перевода:** значение переводится в доменный формат, если его потребляет доменная или UI-логика; значение остаётся во внешнем формате, если оно лишь пересекает приложение от одной внешней границы к другой (например, данные, которые только показываются модели и не участвуют в доменной логике).
- **Тип с несколькими потребителями.** Если значение идёт и в домен/UI, и к модели, оно хранится во внутреннем формате (`camelCase`), а на LLM-границе отдаётся snake-проекцией. Формат не выбирается «по большинству полей» или «по меньшему объёму правок»: как только данные потребляет UI или домен, внутренний тип становится `camelCase`, а граница проецирует.
- **LLM-контракт намеренно остаётся во внешнем формате.** Имена инструментов и поля их входов держим в том виде, который привычен модели и стабилен как интерфейс. Этот формат заперт в слое инструментов и схем и глубже в домен не проникает.

## 3. Источник истины для типов

- **Неконтролируемый рантайм-вход валидируется схемой; тип выводится из схемы.** Тип не пишется руками параллельно валидации. Сюда относятся: тело входящих HTTP-запросов, данные от модели, переменные окружения.
- **Внутренние доменные структуры описываются рукописными типами.** Если структура нигде не парсится и существует только внутри приложения — ей не нужна схема.
- **Полудоверенные границы.** Когда внешний источник доступен через типизированную клиентскую библиотеку, допустимо доверять её типам и сужать без рантайм-валидации — при условии, что риск расхождения и радиус последствий малы. Строгую рантайм-валидацию резервируем для реально неуправляемых источников.
- **Одна сущность — один тип на состояние.** Пустые алиасы не заводим. Если два формально разных типа структурно идентичны — это один тип.

## 4. AI/LLM-слой

- **Промпты, определения инструментов и конфигурация провайдера — отдельные модули,** а не инлайн в роуте или оркестраторе.
- **Провайдер модели — за абстракцией.** Выбор модели и провайдера сосредоточен в одной точке, что позволяет менять их и поддерживать fallback без правок по всему коду.
- **Инструменты проектируются как интерфейс для модели:** семантичные namespaced-имена, понятные имена полей, error-сообщения как actionable-подсказки, схема входа.
- **Контракт стрима между бэкендом и фронтендом — единый источник,** из которого выводятся обе стороны. Тип сообщения не дублируется.
- **Проверенную механику не переписываем.** Если паттерн стриминга подтверждён как канонический для текущего стека — он остаётся; меняется только размещение файлов.

## 5. Оркестрация и роуты

- **Route handler тонкий:** валидирует вход, делегирует доменному оркестратору, возвращает результат. Бизнес-логика, лимиты, выбор провайдера или мок-режима — в доменном слое, не в роуте.
- **Серверная логика помечается явно.** Модули, работающие с секретами и внешними клиентами, изолируются барьером, который не даёт серверному коду утечь в клиентский бандл.
- **Конфигурация фичи централизована.** Лимиты и пороги живут в одном месте, а не разбросаны по коду константами.

## 6. Конфигурация и окружение

- **Всё, что меняется между деплоями** (секреты, ключи, хосты, флаги), — через переменные окружения, не в коде.
- **Окружение валидируется на старте** типизированной схемой по принципу fail-fast: при отсутствии или неверном значении приложение падает сразу с понятной ошибкой, а не глубже по ходу работы.
- **Внешнюю зависимость под конфигурацию не вводим,** пока рукописное решение покрывает потребность.

## 7. Нейминг

- **Файлы и папки** — `kebab-case`.
- **Типы и интерфейсы** — `PascalCase`, без префикса `I`.
- **Поля** — `camelCase` в домене и UI; `snake_case` только на внешних границах.
- **Компоненты** — файл в `kebab-case`, экспорт в `PascalCase`. **Хуки** — префикс `use`.
- **Фабрики** — единый префикс (`create`).
- **Суффиксы по роли файла** (например, для схем) — выборочно, только там, где роль файла иначе неочевидна. Co-location по умолчанию, без сплошного выноса типов в отдельные файлы.
- **Константы** — `UPPER_SNAKE_CASE`. **Имена инструментов модели** — `snake_case`, namespaced.

## 8. Что осознанно не делаем

Анти-оверинжиниринг для проекта текущего размера (одна доминирующая фича):

- **Не вводим** `features/` или `src/` — для одно-фичевого приложения это лишний слой.
- **Не строим полный hexagon / DDD** со слоями use-cases и доменными классами с поведением: у сущности нет инвариантов, это data-in / data-out. Порты заводим только для внешних сервисов.
- **Не централизуем типы** в общую папку — co-location рядом с кодом.
- **Не плодим barrel-файлы** в клиентских путях; они допустимы только как публичный вход server-only модуля.
- **Не добавляем абстракции и зависимости авансом** — вводим их, когда простое решение перестаёт справляться.
